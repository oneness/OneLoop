Core Idea

Directives Are a Pre-Processing Layer

Directives are not sent to the model. They are parsed from the user's input before any provider call to decide which provider(s) to use and how to orchestrate them. The model never sees #!.

User types input
→
Parse #! directives
→
Select provider(s)
→
Send prompt body to model(s)
→
Return result

Key Principle

Directives control runtime orchestration. Skills control model behavior. A skill teaches the agent how to review code; a directive decides which providers review it. They never overlap.

I'd recommend Lua plugins first. The REPL-driven development cycle is faster for iteration, and the lifecycle hooks (on_tool_call, on_compact) can be tested immediately without restarting oneloop…

Recommendation: Ship markdown skills first, then Lua plugins.

All three models agree that skills offer lower friction for the majority of users. Anthropic and Z.AI emphasize the zero-setup nature of dropping .md files; OpenAI highlights the faster iteration cycle of Lua. The practical tradeoff is that skills can't intercept tool calls or lifecycle events — that's where Lua fills the gap. Suggested path: skills loader → safety plugin (Lua) → memory plugin (Lua).
oneloop — Directive System 5
Mode 3

Debate — Critique, Revise, Then Decide

Like consensus, but with structured critique/revision rounds before the final synthesis. Each provider sees the others' answers and is forced to identify weaknesses in their own reasoning. Controlled by #!rounds (default 1, max 3).

Round 1 — Initial Answers
All providers answer independently in parallel. No cross-contamination.
Round 2 — Critique / Revision
Each provider receives the full transcript and must critique the others, identify gaps in their own reasoning, and provide a revised position.
Round 3 (if #!rounds 2) — Deepened Critique
Second critique round using all prior critiques and revisions. Forces deeper engagement with counterarguments.
Final — Judge Synthesis
Judge provider reads the full debate transcript and produces the final consensus. 
#!debate anthropic openai zai
#!rounds 2
#!judge anthropic

pub fn parse_prompt(input: &str) → Result<PromptDirectives> {
    // No #! → default single mode, full prompt is body
    if !input.starts_with("#!") { return default_single(input); }

    // Walk lines, parse directive block, collect body
    for line in lines {
        if starts_with("#!") → parse_directive_line()
        else          → body
    }

    validate(&state)  // one pass: judge/rounds vs mode compatibility
    return PromptDirectives { mode, judge, rounds, tools, prompt }
}

split_providers(tokens)

The one helper. Greedily collects known provider names from the front of a token list. Returns (providers, remaining_body). Used by #!provider, #!consensus, #!debate, and provider shorthand — all four paths share the same logic.

State::set_mode()

Rejects mode conflicts at insertion time. If mode is already set, bails immediately. Eliminates the old resolve_mode() pass — if parsing succeeds, the mode is already resolved. No post-hoc reconciliation needed.
