Bug & Warnings

BUG Empty assistant text blocks crash Anthropic API

src/agent.rs:537 src/providers/anthropic.rs:213

When Anthropic responds with tool_calls but no text content (common for Claude models), the orchestration evidence loop sends an AssistantMessage { content: "" }. Anthropic's API rejects empty text blocks: "text content blocks must be non-empty".

// agent.rs:537 — always appends, even if content is ""
req.messages.push(Message::Assistant(AssistantMessage {
    content: response.content.clone(),  // can be ""
}));

Fix: Skip the Assistant message when content is empty and tool calls are present. Anthropic allows assistant messages with only tool_use blocks.

WARN Main agent loop has the same empty-content risk

src/agent.rs:714

In run_once_with, empty assistant content is guarded (if !response.content.trim().is_empty()), but the guard only controls printing and session storage. If a future change sends the message history back to Anthropic directly (not via session replay), the same crash would occur. The guard is correct today but fragile.

WARN Evidence cache uses std::sync::Mutex in async context

src/agent/evidence.rs:16

The EvidenceCache uses std::sync::Mutex which is fine for quick lock/unlock, but execute() calls cache.lock().unwrap().has() and then drops the lock before doing async work. If someone adds a call that holds the lock across an .await, it will deadlock silently. Consider documenting the contract or switching to tokio::sync::Mutex.

WARN Tool definitions hardcoded in evidence tool schema

src/agent/evidence.rs:72-95

The request_evidence tool definition lists available tools as a hardcoded enum ("read", "web_search", "shell") in its JSON schema. If tools are added or removed, this must be manually synced. Should derive from the allowed_tools() set or the ToolRegistry.

Suggestions

SUGGEST Format directives are parsed but never applied

src/directives.rs src/app.rs

The parser supports format:md and format:html and stores them in PromptDirectives.format. But run_directives() in app.rs never reads directives.format — the output format field is ignored entirely. Either wire it up or remove it from the parser to avoid misleading users.

SUGGEST Orchestration doesn't propagate errors to the user

src/agent.rs:468

If a provider task panics or fails inside the spawned orchestration tasks, the error surfaces as "provider task failed" with a tokio JoinError. The original error (e.g. Anthropic 400) is swallowed. Consider logging the per-provider error before the generic message.

SUGGEST Debate transcript grows unbounded in context

src/agent.rs:383

Each debate round calls format_debate_round_prompt() which includes the full transcript. With 3 providers × 3 rounds, this can be very large. Consider truncating earlier rounds or only sending the latest N responses.

SUGGEST No request timeout on provider calls

src/providers/anthropic.rs src/providers/openai.rs src/providers/zai.rs

None of the provider HTTP clients set a request timeout. A hung provider connection will block the agent indefinitely (until the retry logic kicks in on the next attempt). Consider adding a 60-120s timeout to each reqwest::Client.

Verdict

The codebase is well-structured for its size. One real bug (empty Anthropic text blocks) that should be fixed immediately — it will crash any orchestration where Claude responds with tool calls only. The warnings are low-risk but worth addressing. Suggestions are quality-of-life improvements for later.
